# 目标
把 PDF / PPT 导出改成“所见即所得”的高清截图模式，确保导出页面与网页浏览页面保持一致，不再依赖会导致错位、乱码、拉伸的 DOM 重排或文本重建。

# 计划
1. 重构导出渲染链路
   - 只保留“真实页面截图”作为导出源。
   - 每一页都从 `/print/:n` 的真实网页视图生成固定分辨率截图，作为 PDF 和 PPT 的唯一内容来源。
   - 去掉任何可能重新排版文本或单独拼装元素的中间逻辑，避免再次回到旧方案。

2. 强化截图前的“页面就绪”判定
   - 在打印页中等待以下条件全部满足后再截图：内容 overrides 已加载、字体加载完成、图片加载完成、连续两帧稳定渲染完成。
   - 增加失败保护和超时提示，避免页面还没稳定就开始抓图。
   - 重点检查中文字体和替换图片，减少乱码和尺寸错位。

3. 提升截图质量与一致性
   - 固定导出视口为 1920×1080，并提高截图采样倍率，确保清晰度。
   - 统一 PDF 与 PPT 都使用同一批截图，不再走两套不同渲染结果。
   - 校正 iframe / body / slide 容器尺寸与背景，避免边缘裁切、透明背景或拉伸。

4. 彻底隔离旧缓存结果
   - 更新导出版本标识与内容哈希策略，强制让新截图方案生成全新的缓存文件路径。
   - 下载时优先获取当前新版本缓存，避免历史异常文件再次被命中。
   - 保留后台预生成，但只预生成新的截图制品。

5. 优化前端状态反馈
   - 导出和静默预生成阶段明确显示为“正在生成高清截图”“正在合成 PDF/PPT”等状态。
   - 当用户点击下载时，如果后台已有同版本生成任务，则直接提示等待后台高清版本完成。

6. 验证结果
   - 检查关键页面导出结果是否与网页页面一致。
   - 重点核对中文文本、图片比例、旋转装饰、复杂布局页是否仍出现错位或乱码。
   - 若发现个别页面仍不稳定，再针对打印页样式做最小范围修正，而不是回退导出方案。

# 技术说明
- 现状里已经有 `/print/:n` 路由和截图式导出的基础，但还需要进一步“收敛”为唯一可信链路，并把旧缓存/旧渲染影响彻底切掉。
- 现有后端已具备 `export_cache` 与导出文件存储能力，因此这次不需要新增数据库表，主要是前端导出流程与缓存版本策略调整。
- 修改范围预计集中在：
  - `src/lib/export.tsx`
  - `src/pages/Print.tsx`
  - `src/pages/Index.tsx`
  - 如有必要，少量校正与打印态相关的 slide/container 样式文件

# 预期结果
导出的 PPT 和 PDF 会变成每页一张高清网页截图：视觉上与网页看到的页面一致，基本杜绝文字乱码、局部错位、布局被重新拉伸的问题。
